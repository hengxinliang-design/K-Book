# K-Book 二次开发需求基线

## 1. 文档状态

- 状态：已确认
- 基线项目：Open Notebook
- 基线提交：`14ba8f51e81f34855cd21c390f2576215d8808dd`
- 目标：在保留 Open Notebook 文档处理、检索和问答能力的基础上，增强文件组织与知识库管理能力。

本文档是 K-Book 二次开发的需求边界。未列入本文档的业务流程，不在当前开发范围内。

## 2. 产品边界

K-Book 不实现客服工单、问题转派、顾问审批流程或项目实施流程。

K-Book 当前负责：

1. 创建和管理知识库。
2. 上传、替换和重新学习文件。
3. 按目录、标签和业务元数据组织文件。
4. 对知识库文件进行检索和基于来源的问答。
5. 为后续用户、角色和项目权限提供扩展基础。

## 3. 已确认需求

### KBR-001：逻辑目录层级

使用者可以在知识库内建立多层目录，并使用目录组织文件的呈现。

要求：

- 支持创建、重命名、移动和删除目录。
- 支持任意合理层级的父子目录。
- 文件可以移动到不同目录。
- 删除非空目录时必须明确处理其子目录和文件。
- 目录是逻辑组织结构，不改变底层原始文件内容。
- 文件检索不应因目录层级而失效。

验收：

- 使用者能够通过目录树浏览文件。
- 同一个知识库内可以建立多级目录。
- 文件移动后仍可正常查看、检索和问答。

### KBR-002：文件标签

在现有 `topics` 能力基础上，提供面向使用者的标签管理。

要求：

- 一个文件可以关联多个标签。
- 标签可以添加、移除、搜索和筛选。
- 标签名称来自统一标签字典，不允许无约束地产生重复近义标签。
- 标签与目录相互独立：目录用于层级组织，标签用于多维分类。

验收：

- 使用者能够按一个或多个标签筛选文件。
- 标签修改不触发文档重新学习。
- 标签删除前能够看到受影响文件。

### KBR-003：文件业务元数据

文件需要支持以下可管理元数据：

| 元数据 | 说明 |
| --- | --- |
| 模块 | 文件所属 Infor LN 业务或产品模块 |
| 文档类型 | 需求、蓝图、产品设计、方案设计、技术设计、代码设计等 |
| 版本 | 文件自身的业务版本 |
| 状态 | 文件当前使用状态 |

要求：

- 元数据可以在上传时填写，也可以上传后修改。
- 元数据用于列表展示、组合筛选和检索范围约束。
- 元数据修改不改变原始文件，也不必重新生成向量。
- 元数据选项应由管理配置维护，避免自由文本失控。

具体枚举值在后续信息架构设计中确认。

### KBR-004：知识库属性

每个知识库除现有名称和描述外，还需要定义适用范围。

知识库属性至少包括：

| 属性 | 说明 |
| --- | --- |
| 名称 | 知识库显示名称 |
| 描述 | 知识库用途说明 |
| 客户 | 适用客户 |
| 项目 | 适用项目 |
| LN 版本 | 适用的 Infor LN 版本 |
| 适用范围 | 对知识库边界和使用条件的文字说明 |

要求：

- 创建知识库时提供属性填写步骤。
- 知识库属性可以后续修改。
- 客户、项目和 LN 版本可以用于知识库筛选。
- 问答界面应明确显示当前知识库的适用范围。

### KBR-005：统一标签字典

系统提供统一标签字典，集中管理可用标签。

要求：

- 支持创建、重命名、停用和删除标签。
- 标签至少具有名称、状态和说明。
- 标签名称在有效范围内唯一。
- 停用标签不允许继续关联新文件，但保留历史关联。
- 删除已使用标签时必须进行影响确认。
- 文件上传和编辑时从标签字典选择标签。

标签字典和模块、文档类型、状态等业务字典应采用一致的管理交互，但它们是不同的字段类型。

### KBR-006：上传分类与命名步骤

现有上传流程需要增加分类和命名步骤。

目标流程：

```text
选择文件
→ 选择知识库和目录
→ 确认显示名称
→ 设置标签和业务元数据
→ 确认处理选项
→ 上传、解析和学习
```

要求：

- 默认使用原文件名生成显示名称。
- 使用者可以在上传前修改显示名称。
- 批量上传时支持统一设置知识库、目录和公共元数据。
- 批量上传时仍允许逐文件修改名称和元数据。
- 上传确认页展示最终文件名称、目录、标签和元数据。
- 文件显示名称与底层存储文件名分离。

命名规则的自动生成方式在后续交互设计中确认。

### KBR-007：替换文档并重新学习

允许使用新文件替换已经上传的文档，并重新执行内容解析、分块和向量化。

要求：

- 替换操作保留原来源记录的业务身份和知识库关联。
- 默认保留显示名称、目录、标签和业务元数据。
- 新文件成功处理前，旧版本仍保持可用。
- 新文件解析和学习成功后，原检索内容与向量整体切换为新版本。
- 处理失败时回滚到旧版本，不得留下部分新索引。
- 记录替换时间、原文件信息、替换文件信息和处理结果。
- 替换期间在界面显示处理状态。
- 支持重新学习当前文件，而无需重新上传。

一期至少保留最近一次可回滚版本；完整版本历史能力后续确认。

### KBR-008：用户、角色及项目权限

用户、角色及项目权限属于后续设计范围，不在第一期直接实现。

当前设计必须避免阻塞后续权限扩展：

- 目录、文件、标签和知识库均应具有稳定标识。
- 客户和项目不能只作为显示文本处理。
- API 不应把当前单用户假设写死到新增领域模型中。
- 后续需要支持按知识库、客户、项目和文件控制访问。

正式的身份认证、角色模型、授权规则和权限继承将在独立阶段设计。

### KBR-009：屏蔽 Podcast

K-Book 不提供 Podcast 功能。

一期要求：

- 从主导航中移除 Podcast 入口。
- 从快速创建菜单中移除 Podcast。
- Podcast 页面不可通过正常产品导航访问。
- 模型设置中不展示仅供 Podcast 使用的配置。
- K-Book 文档和界面中不宣传 Podcast 能力。

为降低与 Open Notebook 上游同步的成本，一期可以保留后端 Podcast 源代码和数据库迁移，但 K-Book 产品界面和正常使用路径必须完全屏蔽。是否彻底删除代码在后续维护阶段决定。

## 4. 数据设计原则

现有 `Source.title` 和 `Source.topics` 保留，不以大量新增字段直接挤入 `Source` 作为唯一实现方式。

逻辑目录、统一字典、业务元数据和文件版本应形成独立、可关联的领域对象。这样可以：

- 保持 Open Notebook 核心来源模型稳定。
- 降低合并上游更新时的冲突。
- 支持统一字典和后续权限控制。
- 支持文件替换及版本审计。
- 避免将目录路径、客户和项目变成不可治理的自由文本。

具体表结构和关系模型将在实现设计阶段确认。

## 5. 开发优先级

### 第一期：文件组织与管理

1. 屏蔽 Podcast 产品入口。
2. 知识库属性：名称、客户、项目、LN 版本和适用范围。
3. 统一标签及业务字典。
4. 逻辑目录层级。
5. 文件模块、文档类型、版本和状态元数据。
6. 新上传分类及命名流程。
7. 文件列表的目录浏览、标签及元数据筛选。

### 第二期：文件生命周期

1. 替换文件。
2. 安全重新学习。
3. 处理失败回滚。
4. 基础版本和替换记录。

### 后续阶段：权限

1. 用户与身份认证。
2. 角色和权限。
3. 客户及项目成员关系。
4. 知识库、目录和文件访问控制。
5. 审计日志。

## 6. 明确不做

当前范围不包括：

- 客服工单管理。
- 问题转派和审批流程。
- 项目实施过程管理。
- 使用自由文本代替统一标签和业务字典。
- Podcast 产品能力。
- 第一期实现完整企业级用户权限。
